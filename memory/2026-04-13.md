# 2026-04-13 - Catch-RedA 项目产品文档分析

## 最终分析报告

### 一、项目概况总结

#### 1.1 项目基本信息
- **产品名称**: Catch RedA
- **运行平台**: 微信小程序
- **对局人数**: 2-4 人
- **屏幕方向**: 横屏
- **单局目标时长**: 
  - 2 人局：3-6 分钟
  - 3 人局：5-8 分钟
  - 4 人局：6-10 分钟

#### 1.2 核心玩法
- 使用标准 54 张扑克牌去掉方块 A，保留红桃 A（共 53 张）
- 顺时针从下一家抽牌
- 形成对子（点数相同且颜色相同）后自动消除
- 最终持有红桃 A 的玩家失败
- 核心竞争点：记忆博弈

#### 1.3 技术架构
- **前端**: 微信小程序（兼容 iOS/Android 主流微信版本）
- **后端**: HTTP + WebSocket 实时同步
- **状态维护**: 服务端统一维护，客户端只做展示
- **数据一致性**: 服务端事务保证，支持断线重连

#### 1.4 核心模块

**模块一：大厅与房间系统**
- 房间列表、搜索、匹配、创建房间
- 好友列表（微信好友 + 游戏好友）
- 邀请与追踪系统
- 房间状态管理（waiting/countdown/selecting_role/starting_game/playing/settlement/closed）
- 房主权限管理（踢人、关位、开位、解散）

**模块二：对局系统**
- 发牌规则（不同人数发牌数量不同）
- 自动消牌逻辑
- 回合抽牌机制
- 技能触发与结算
- 出牌与名次判定
- 金币结算

**模块三：角色技能皮肤系统**
- 首发 2 个初始角色：街溜子（排序技能）、小学生（透视技能）
- 技能系统：主动技能、被动技能
- 皮肤系统：立绘、动画、语音
- 金币购买与抽卡系统

**模块四：客户端交互与表现**
- 第一人称圆桌视角
- 抽牌、消牌、结算动画
- 弱网降级处理
- 断线重连机制
- 音效与语音

**模块五：服务端数据结构**
- 用户 User
- 房间 Room
- 房间成员 RoomMember
- 对局 Game
- 技能状态 GameSkillState
- 手牌 GameCard
- 资产流水 GoldLedger

**模块六：战绩与称号系统**
- 胜率统计
- 连胜记录
- 6 个称号阶段 + 星级系统
- 称号晋升规则

#### 1.5 技术栈总结
- **前端**: 微信小程序原生开发
- **通信**: HTTP + WebSocket
- **状态管理**: 服务端权威数据 + 客户端缓存
- **数据持久化**: 服务端存储（未指定具体方案）

### 四、关键决策点

1. **架构决策**: 采用前后端分离还是单体架构？
   - 建议：单体架构（便于快速开发迭代）或微服务（便于扩展）

2. **数据库选型**: MySQL 或 MongoDB？
   - 建议：MySQL（关系型数据为主）+ Redis（缓存）

3. **WebSocket 服务**: 自行实现或使用第三方 SDK？
   - 建议：使用成熟的 WebSocket 服务

4. **存储策略**: 本地存储还是云存储？
   - 建议：云存储（便于多端同步）
